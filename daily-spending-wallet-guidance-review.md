# Oubli vs Bitcoin Design "Daily Spending Wallet" Guidance

## Short Take

Against the [Bitcoin Design daily spending wallet](https://bitcoin.design/guide/daily-spending-wallet/) reference, Oubli follows the basics of a mobile self-custodial wallet reasonably well, but it does not yet match the guide's stronger patterns around contacts, metadata, backup/recovery, privacy defaults, and safe payment confirmation. Some divergence is intentional because Oubli is not a pure bitcoin-only wallet; other gaps are clear UX misses.

## Where Oubli Follows The Guidance

- Oubli clearly presents itself as self-custodial and forces the user through create-or-restore onboarding on both platforms, which matches the guide's emphasis on first-use clarity and key ownership. See [OnboardingView.swift](ios/Oubli/Views/OnboardingView.swift) and [OnboardingScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/OnboardingScreen.kt). This aligns with the guide's [first-use](https://bitcoin.design/guide/daily-spending-wallet/first-use/) direction.
- Send and Receive are primary actions on the home screen on both apps, which is consistent with the guide's preference for fast payment entry from the main screen. See [BalanceView.swift](ios/Oubli/Views/BalanceView.swift) and [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt). This matches [sending](https://bitcoin.design/guide/daily-spending-wallet/sending/) and [requesting](https://bitcoin.design/guide/daily-spending-wallet/requesting/).
- Oubli supports several payment entry methods the guide expects: manual entry, paste, QR scan, and saved contacts. See [SendSheet.swift](ios/Oubli/Views/SendSheet.swift), [ContactPickerView.swift](ios/Oubli/Views/ContactPickerView.swift), [SendDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/SendDialog.kt), and [ContactListDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/ContactListDialog.kt). That is in line with the guide's recommended [payment entry points](https://bitcoin.design/guide/daily-spending-wallet/sending/).
- Both platforms have an app-level biometric barrier, which aligns with the guide's baseline wallet security model. See [LockedView.swift](ios/Oubli/Views/LockedView.swift) and [LockedScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/LockedScreen.kt). This maps to [security](https://bitcoin.design/guide/daily-spending-wallet/security/).
- Activity uses user-facing labels like `Received`, `Sent`, `Settled`, and `Emergency Exit` instead of raw event names, which is directionally aligned with the guide's advice to prefer meaning over protocol detail. See [BalanceView.swift](ios/Oubli/Views/BalanceView.swift) and [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt). This fits the spirit of [activity](https://bitcoin.design/guide/daily-spending-wallet/activity/).
- Receive is explicit on both platforms, with separate Oubli, Starknet, and Lightning modes plus share-sheet support. See [ReceiveSheet.swift](ios/Oubli/Views/ReceiveSheet.swift) and [ReceiveDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/ReceiveDialog.kt). That is at least consistent with the guide's expectation that request flows make formats understandable.

## Where Oubli Does Not Follow The Guidance

- Oubli now has a contact system on both platforms, but it is still lighter than the guide’s richer contact model. Contacts exist for picking recipients and labeling some activity, yet they do not carry the broader metadata, avatars, grouping, or deeper request integration the guide expects. See [ContactListView.swift](ios/Oubli/Views/ContactListView.swift), [ContactDetailSheet.swift](ios/Oubli/Views/ContactDetailSheet.swift), [ContactListDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/ContactListDialog.kt), and compare with [contacts](https://bitcoin.design/guide/daily-spending-wallet/contacts/).
- Oubli still lacks metadata in many places the guide expects it. Contacts can store notes, but send/request flows still do not capture payer names, labels, tags, or rich descriptions, and activity remains relatively thin. See [SendSheet.swift](ios/Oubli/Views/SendSheet.swift), [BalanceView.swift](ios/Oubli/Views/BalanceView.swift), [SendDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/SendDialog.kt), and [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt). That falls short of [sending](https://bitcoin.design/guide/daily-spending-wallet/sending/), [requesting](https://bitcoin.design/guide/daily-spending-wallet/requesting/), and [activity](https://bitcoin.design/guide/daily-spending-wallet/activity/).
- Activity is still much thinner than the reference design. There is no search, filtering, export, or advanced grouping, even though both apps now have transaction detail views and contact-name lookups. See [BalanceView.swift](ios/Oubli/Views/BalanceView.swift), [ActivityDetailSheet](ios/Oubli/Views/BalanceView.swift), [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt), and compare with [activity](https://bitcoin.design/guide/daily-spending-wallet/activity/).
- Privacy only partially follows the guide. Oubli does let users hide the balance, but it is visible by default and the toggle is still a tap on the whole card, with no hold-to-reveal, no PIN-to-reveal option, and no privacy settings surface. See [BalanceView.swift](ios/Oubli/Views/BalanceView.swift) and [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt). The guide's [privacy](https://bitcoin.design/guide/daily-spending-wallet/privacy/) and [requesting](https://bitcoin.design/guide/daily-spending-wallet/requesting/) guidance is stronger than what Oubli ships.
- Backup/recovery is still materially behind the reference design. Oubli exposes the seed phrase during onboarding and supports restore from phrase, but the iOS backup flow remains limited and Android backup/reveal still do not reach the guide’s stronger posture around reminders, safer recovery UX, and cloud-backed recovery. See [OnboardingView.swift](ios/Oubli/Views/OnboardingView.swift), [OnboardingScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/OnboardingScreen.kt), [SeedBackupView.swift](ios/Oubli/Views/SeedBackupView.swift), [MainScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/MainScreen.kt), and [WalletViewModel.kt](android/app/src/main/java/com/oubli/wallet/viewmodel/WalletViewModel.kt). I also did not find user-facing cloud backup, first-funds backup reminders, or recovery of metadata/history as described in [backup & recovery](https://bitcoin.design/guide/daily-spending-wallet/backup-and-recovery/landing-page/) and [recovery](https://bitcoin.design/guide/daily-spending-wallet/backup-and-recovery/recovery/).
- Settings remain too thin relative to the guide. Both apps expose useful actions such as `Contacts`, fiat currency selection, and `Show Seed Phrase`, but they still lack a dedicated grouped settings surface for privacy, security, backup, network, and help. See [BalanceView.swift](ios/Oubli/Views/BalanceView.swift) and [BalanceScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/BalanceScreen.kt). The guide expects a much fuller [settings](https://bitcoin.design/guide/daily-spending-wallet/settings/) experience.
- Security is only baseline-complete. Biometrics exist, but I did not find PIN protection, security checklist/recommendations, wallet limits, or reminder flows triggered by deposits or risk changes. See [LockedView.swift](ios/Oubli/Views/LockedView.swift), [LockedScreen.kt](android/app/src/main/java/com/oubli/wallet/ui/LockedScreen.kt), and compare with [security](https://bitcoin.design/guide/daily-spending-wallet/security/).
- Oubli still blocks users inside processing states more than the guide recommends. The guide says users should not be forced to wait in a blocked interface while a transaction completes. Send flows in particular still disable dismissal during Lightning processing on both platforms, and iOS receive still blocks dismissal while waiting. See [SendSheet.swift](ios/Oubli/Views/SendSheet.swift), [ReceiveSheet.swift](ios/Oubli/Views/ReceiveSheet.swift), and [SendDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/SendDialog.kt). This conflicts with [sending](https://bitcoin.design/guide/daily-spending-wallet/sending/).
- Receive sharing is stronger than in the original review because both platforms now expose share-sheet actions for Oubli, Starknet, and Lightning requests. The remaining gap is richer request metadata and broader delivery patterns rather than the absence of sharing itself. See [ReceiveSheet.swift](ios/Oubli/Views/ReceiveSheet.swift) and [ReceiveDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/ReceiveDialog.kt). That still falls short of the fuller request model in [requesting](https://bitcoin.design/guide/daily-spending-wallet/requesting/).

## By-Design Differences

- The reference design is bitcoin-only and Lightning-first. Oubli is not that product. Its receive model explicitly includes `Oubli`, `Starknet`, and `Lightning`, and Lightning appears to bridge into a privacy pool rather than act as the wallet's single native balance. That is visible in [ReceiveSheet.swift](ios/Oubli/Views/ReceiveSheet.swift) and [ReceiveDialog.kt](android/app/src/main/java/com/oubli/wallet/ui/balance/ReceiveDialog.kt). So some mismatch is strategic, not necessarily wrong, but it means Oubli should not assume the reference design fully fits without adaptation.

## Bottom Line

Oubli follows the guide on core wallet scaffolding: self-custody messaging, home-screen send/receive, biometric lock, and basic payment entry. It falls short on the parts that make a daily-spending wallet feel mature and trustworthy in daily use: safe invoice import, contacts, metadata, richer activity, privacy defaults, reliable backup/recovery, and a real settings/security surface.
